## 4. 問題清單（依嚴重度分級）

### 🔴 P0 — 阻斷流程或資安（必須先修）

#### ① 後端授權檢查完全缺失（資料外洩）
- **檔案**：`backend/app/services/session_service.py:161-180` (`get_by_id`) / `460-483` (`get_session` / `list_sessions` / `get_conversations`)
- **問題**：所有讀取方法都只 `WHERE Session.id == session_id`，完全沒有檢查 `current_user` 是否擁有該場次。雖然簽名裡收了 `current_user` 參數，但完全未使用。
- **影響**：登入的病患 A 只要知道 session UUID，就能讀到病患 B 的完整問診逐字稿、紅旗紀錄、診斷資訊。HIPAA / 醫療隱私法違規。
- **修復成本**：低（4-5 個方法加 ownership filter）
- **狀態**：🟡 待修復

#### ② WebSocket token 放在 query string
- **檔案**：`frontend/src/services/websocket.ts:100-102` / `backend/app/websocket/conversation_handler.py:252`
- **問題**：`fullUrl = ${url}?token=${token}`，JWT 明文附在 WebSocket URL。
- **暴露面**：Railway access log、Vercel edge log、瀏覽器歷史、各層 proxy、Sentry breadcrumb、任何開發者工具。
- **修復方向**：
  - 方案 A：連線後首條 `auth` frame 送 token（server 在收到 auth 前保持連線但不處理其他訊息）
  - 方案 B：`Sec-WebSocket-Protocol: Bearer, <token>` subprotocol
- **修復成本**：中（前後端協同）
- **狀態**：🟡 待修復（放 Wave 4，不急）

#### ③ `backend/.env` 的 `CORS_ORIGINS` 不含 Vercel domain（本機開發用）
- **檔案**：`backend/.env:9` / `backend/app/main.py:46-52` / `backend/app/core/config.py:154-157`
- **問題**：本機 `.env` 的 `CORS_ORIGINS` 只有 `localhost:5173` / `localhost:3000`，沒有 Vercel domain。
- **現況（2026-04-15 已核對 Railway Raw Editor）**：Railway production 環境變數 `CORS_ORIGINS` 已正確設為 `["https://gu-voice-jht12020304y-7696s-projects.vercel.app","http://localhost:3000","http://localhost:5173"]`，**production CORS 功能正常**（見下方 ⑰ 改寫紀錄）。
- **影響**：僅影響本機跑 `uvicorn` 時若前端用 production Vercel URL 連本機 API 的情況，實務上不會發生（開發都是 localhost↔localhost）。
- **修復成本**：無需修復（本機 `.env` 保持 localhost 即可）
- **狀態**：🟢 非問題（已澄清）

#### ⑰ ~~`/auth/refresh` CORS 全破~~ **已撤回 — 誤判**
- **原判定（2026-04-15 上午）**：以為 `/auth/refresh` preflight 被 CORS 擋，根據 Playwright F12 console 錯誤訊息與直接用 curl 打 `OPTIONS /auth/refresh` 回 405。
- **撤回原因（2026-04-15 下午複測）**：我的 probe 漏了 `/api/v1` 路徑前綴。對正確路徑重測：
  ```
  OPTIONS /api/v1/auth/refresh  → 200 + Access-Control-Allow-Origin: https://gu-voice-jht12020304y-7696s-projects.vercel.app
  OPTIONS /api/v1/auth/login    → 200
  OPTIONS /api/v1/sessions      → 200
  GET /api/v1/health (Origin)   → Access-Control-Allow-Origin 存在
  ```
- **F12 的錯誤訊息解讀**：那個 console error 很可能是瀏覽器在 `/auth/me` 401 → `/auth/refresh` 拋 401（不是 CORS）後 axios interceptor 在沒有 refresh_token 時的正常登出流程，被 Chrome devtools 顯示成看似 CORS 的錯誤。實際 response header 都有 ACAO。
- **狀態**：🟢 **撤回**（CORS 功能正常）

### 🟠 P1 — 功能正確性與穩定性

#### ④ Session status 變更 race condition
- **檔案**：`backend/app/websocket/conversation_handler.py:429-473`（idle watchdog） vs `500-538`（`end_session` control）
- **問題**：兩個 async 任務同時呼叫 `_update_session_status`，沒有 optimistic lock 或共享 `finished` 旗標。`VALID_TRANSITIONS` 會擋下第二個 commit 拋 exception，但 Redis/DB 是否同步需要驗證。
- **修復**：加 `asyncio.Lock` 或共享 `finished` 旗標
- **狀態**：🟡 待修復

#### ⑤ Session 完成的非原子性
- **檔案**：`backend/app/websocket/conversation_handler.py:1587-1635` 附近
- **問題**：SOAP 生成、DB 寫入、Redis 狀態更新三者不是原子操作，失敗可能導致部分狀態寫入
- **狀態**：🟡 待修復（與 ④ 一起修）

#### ⑥ `create_session` 的 get-or-create patient 漏洞
- **檔案**：`backend/app/services/session_service.py:352-394`, L390/L416
- **問題**：
  1. 若 `patient_info` 缺 `date_of_birth` → fallback 用 `date(1900,1,1)`
  2. 之後同一使用者每次都會命中這個 placeholder 而不是新建
  3. DB 沒有 `(user_id, name, dob, phone)` unique 約束 → 重複 POST 會產生重複 Patient
- **修復**：改為要求 `patient_info` 必填關鍵欄位，若缺回 400
- **狀態**：🟡 待修復

#### ⑦ `OPENAI_MODEL_CONVERSATION` 環境不一致 ⭐ 與 gpt-5.4-mini 直接相關
- **檔案**：`backend/app/core/config.py:110` vs `backend/.env`
- **問題**：
  - config.py 預設 `gpt-5.4-mini`
  - backend/.env 覆寫回 `gpt-4o`
  - 結果：本機跑 gpt-4o，Railway 推測跑 gpt-5.4-mini，兩邊不一致
- **修復**：改 backend/.env 或刪除覆蓋行
- **狀態**：🟡 待修復（Wave 1）

#### ⑧ Supervisor 沒設 `max_completion_tokens`
- **檔案**：`backend/app/pipelines/supervisor.py:148-151` 附近 / `backend/app/core/config.py`
- **問題**：config.py 只有 `OPENAI_MAX_TOKENS_CONVERSATION`（2048）和 `OPENAI_MAX_TOKENS_SOAP`（4096），缺 `OPENAI_MAX_TOKENS_SUPERVISOR`。reasoning_effort=medium 時 CoT 輸出可達 5000+ tokens，沒 budget 會被截斷或爆預算。
- **修復**：
  - `config.py` 新增 `OPENAI_MAX_TOKENS_SUPERVISOR: int = 4096`
  - `supervisor.py` 的 `create_kwargs` 加 `max_completion_tokens`
- **狀態**：🟡 待修復（Wave 1）

#### ⑨ STT / LLM / TTS 失敗處理不一致
- **檔案**：`backend/app/websocket/conversation_handler.py:750-900` 附近
- **問題**：
  - STT 失敗只 log warning 不通知前端
  - LLM 失敗直接讓全域 except 把 WS 斷線
  - TTS 失敗前端會收到 `audioB64=None` 但沒有對應 error frame
- **修復**：每種失敗都送 typed `error` frame（`code: "STT_FAILED"` 等），不直接斷線
- **狀態**：🟡 待修復

#### ⑩ Redis context TTL 沒延期
- **檔案**：`backend/app/websocket/conversation_handler.py:39-41`
- **問題**：`_SESSION_CONTEXT_TTL = 3600`（1h），但問診可能超過 1 小時。每次寫 `conversation_history` 時沒 `redis.expire()` 延期 → 90 分鐘的問診會在中途把歷史 GC 掉。
- **修復**：寫入時一併 `await redis.expire(ctx_key, _SESSION_CONTEXT_TTL)`
- **修復成本**：極低
- **狀態**：🟡 待修復

## 6. gpt-5.4-mini 特別提醒（⭐ 使用者明確需求）

### 6.1 關鍵配置表

| 配置 | 建議值 | 原因 |
|---|---|---|
| `OPENAI_MODEL_CONVERSATION` | `gpt-5.4-mini` | ✅ 語音對話主模型 |
| `OPENAI_REASONING_EFFORT_CONVERSATION` | **`none`** ⚠️ | reasoning_effort≠none 時 first-token 延遲 +8~15 秒，會毀掉語音問診的即時感 |
| `OPENAI_MODEL_SUPERVISOR` | `gpt-5.4-mini` | ✅ 背景督導 |
| `OPENAI_REASONING_EFFORT_SUPERVISOR` | `medium` | Supervisor 是背景任務，延遲不影響對話，用 CoT 換督導精準度 |
| `OPENAI_MAX_TOKENS_CONVERSATION` | `2048`（維持） | 語音問診回應短 |
| `OPENAI_MAX_TOKENS_SUPERVISOR` | `4096`（新增） | reasoning medium 的 CoT 輸出會吃掉大量 token |
| `OPENAI_MODEL_SOAP` | `gpt-4o`（維持） | 穩定、便宜 |
| `OPENAI_MODEL_RED_FLAG` | `gpt-4o-mini`（維持） | 穩定、便宜 |

### 6.2 程式碼狀態
`llm_conversation.py:229-234` 和 `supervisor.py:148-151` 的 reasoning_effort / temperature **互斥邏輯已經寫對**：
- `reasoning_effort != "none"` → 送 `reasoning_effort`，不送 `temperature`
- `reasoning_effort == "none"` → 不送 `reasoning_effort`，送 `temperature`

**所以 Conversation + Supervisor 只需要改環境變數，完全不用動程式碼**。

### 6.3 成本與延遲風險
- `gpt-5.4-mini` + `reasoning=medium`：成本約 `gpt-4o` 的 2-3 倍，延遲 +8-12 秒
- 語音問診場景對延遲敏感 → Conversation 維持 `none`，Supervisor 後台用 `medium` 換精準度
- **不要**把 Conversation 改 medium，否則使用者會覺得 AI 每次都「要想很久」

### 6.4 如果切換 SOAP / RedFlag 到 gpt-5.4-mini
若未來想把 SOAP / RedFlag 也切到 gpt-5.4-mini，需要先檢查：
- `backend/app/pipelines/red_flag_detector.py` 是否正確處理 reasoning_effort 切換
- `backend/app/pipelines/soap_generator.py` 同上
- 兩者應套用與 `llm_conversation.py` 相同的互斥邏輯
